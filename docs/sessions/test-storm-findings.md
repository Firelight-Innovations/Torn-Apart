# Test Storm — Suspected-Bug Findings Log

Characterization tests pin **current** behavior at HEAD `8586bb3`. When a worker agent encounters
behavior that looks like a bug, it does NOT fix it and does NOT write the test to the "correct"
answer — it pins the current behavior and reports the suspicion to the orchestrator, who records it
here for human review. (Workers do not edit this file directly, to avoid parallel write collisions.)

Format: `### <module>::<symbol> — <one-line>` then current behavior, why it's suspicious, and the
test that pins it.

---

## Batch 1

### core/clock.py::Clock.fixed_steps — off-by-one at exact N×fixed_dt boundary
Current: accumulating real time in steps of `fixed_dt=0.02` accumulates float error; a frame
delivering exactly `N×fixed_dt` often yields only `N-1` fixed steps (e.g. 5×0.02 → accumulator
~0.019999…993 → 4 steps). Suspicious: a frame worth exactly N ticks of sim time can silently drop a
tick. Pinned in `test_clock.py::TestSpiralOfDeathGuard` / `test_leftover_carried_over`.

### core/clock.py::Clock.update — negative real_dt accepted silently
Current: no guard; negative dt decrements `total_real_time`, accumulator, and `game_time_of_day`
below zero, breaking the `[0, _GAME_SECONDS_PER_DAY)` invariant and rollover logic. Pinned in
`test_clock.py::TestBoundaryInputs::test_negative_dt_pins_current_behaviour`.

### core/lod.py::LODPolicy.band_for — no input validation (4 footguns)
Current, all pinned in `test_lod.py`: (1) unsorted `bands` give wrong band via linear scan;
(2) duplicate thresholds make a band unreachable; (3) empty `bands=()` → every distance is band 0,
no warning; (4) negative distance → band 0 silently. None raise. Likely fine in practice (bands are
constants) but undocumented.

### core/log.py::get_logger — idempotency-guard gap + level override
Current: dual guard (`_handler_installed` flag AND `if not root.handlers`); if another library
installed a root handler before the first `get_logger`, the engine's formatter may never install yet
`_handler_installed` flips True. Also forces root level to WARNING whenever level is NOTSET even if
handlers already exist. `get_logger("")` returns the root logger. Environment-dependent; pinned the
deterministic parts in `test_log.py`.

### terrain/raycast.py::raycast_voxel — inside-solid normal is −direction (non-cardinal)
Current: when origin is inside solid (`last_axis == -1`), `normal = -d` (negated ray dir), not an
axis-aligned face normal. For cardinal rays it coincidentally looks right; for diagonal rays it is a
non-cardinal, and the docstring implies a cardinal normal. Also `max_distance_m=0` still evaluates
the origin voxel (`t>max` checked before first advance). Pinned in `test_raycast.py`.

### terrain/meshing.py::build_mesh — WORLD_FLOOR_SOLID honored for ANY direction
Current: `_build_padded_solid` treats the `WORLD_FLOOR_SOLID` sentinel the same for all 6 neighbor
directions, so passing it for +X/-Y/etc. silently pads that face solid and culls border faces. Only
-Z is documented to accept it; no validation. Pinned in `test_meshing_seams.py`.

### terrain/surface_nets.py::build_mesh_faceted — corner voxel not fully buried by solid neighbors
Current: a single solid voxel at (0,0,0) with all 26 neighbor chunks solid still emits 3 exposed
faces (the inward +X/+Y/+Z faces see the chunk's air interior). Harmless when meshing full chunks
(real callers), but the mesher can't bury a lone corner voxel against solid neighbors. Note:
`face_material` is the solid voxel's own id, direction-independent. Pinned in
`test_surface_nets_correctness.py`.

### terrain/chunk.py::apply_delta — skips baseline generate (doc/code mismatch)
Current: `apply_delta` on an unloaded coord constructs an all-air `Chunk` then overwrites
`materials` from the delta, never calling `generate_chunk` for the baseline. No data lost (the delta
is the full intended array) but contradicts the doc's "generate baseline if needed". Pinned in
`test_chunk_delta.py`.

## Batch 2

### save/save_manager.py::SaveManager.load — non-atomic on apply_delta failure
Current: `clock.set_state()` runs before the per-system `apply_delta` loop. The "no partial load"
guarantee only covers `SaveIncompatibleError` (validated up front); if a system's `apply_delta`
raises mid-loop, the clock (and any already-applied systems) stay mutated. Pinned in
`test_save_manager_edges.py`.

### save/save_manager.py::_decode_value — non-contiguous arrays + read-only decode
Current: `_encode_value` does `obj.tobytes()`; a transposed/non-contiguous view encodes in the
view's logical order and decodes back with the transposed shape (C-contiguous) — a caller expecting
to `.reshape(original_shape)` gets wrong data. Also decoded arrays are `np.frombuffer` read-only, so
an `apply_delta` that writes into the decoded array crashes. Pinned/noted in
`test_save_manager_edges.py`.

### resources/loaders.py::register_loader — silent overwrite + suffix lowercasing
Current: re-registering a suffix silently overwrites with no warning; `register_loader(".PNG")`
stores under `.png` and `dispatch` lowercases the path suffix (case-insensitive). No safeguard
against two registrants colliding. Pinned in `test_resources_loaders.py`.

### zones/grass_placement.py::hash_lowbias32 — 0 is a fixed point (0→0)
Current: lowbias32 has no additive offset, so input 0 maps to output 0; any instance whose
`id ^ seed == 0` collides in placement with every other such instance. Mirrors the GLSL. Pinned in
`test_grass_placement.py`.

### zones/store.py::ZoneStore.get_delta — returns full snapshot before mark_baseline()
Current: with `_baseline is None`, `get_delta()` returns the full volume list (not `{}`), so calling
it before `mark_baseline()` at boot emits unnecessary save data. Also `apply_delta` only ignores
`version > _DELTA_VERSION` (version 0 / missing version are applied — no lower bound). Pinned in
`test_zones_store_edges.py`.

### zones/volume.py::ZoneVolume — ±inf corners accepted; boundaries min-incl/max-excl
Current: validator uses `a < b`, so `(-inf, x)` / `(x, +inf)` corners construct fine (area→inf
downstream). `contains_xy` and `intersects_chunk` are min-inclusive, max-exclusive (a volume whose
max equals a chunk edge does NOT intersect). Pinned in `test_zones_store_edges.py`.

### scene/components.py::coerce_params — non-numeric string raises; case-sensitive; no back-fill
Current: `coerce_params("Light", {"intensity": "abc"})` propagates an uncaught `ValueError` from
`float()`; `"3.5"`→`3.5` accepted; unknown keys dropped; missing keys NOT filled (returns `{}` for
`{}`); `is_known`/catalog lookups are case-sensitive (`"light"`→unknown); unknown kind →`[]` silently.
Relevant if inspector passes untrusted JSON. Pinned in `editor/test_scene_components_coerce.py`.

### scene/runtime.py::SceneRuntime — spawn_position None before rebuild; deferred destroy lingers
Current: `spawn_position` reads `self.objects`, which is empty until `rebuild()` is called, so it
returns `None` even though the store has the spawn. `rebuild()` calls `destroy()` which is deferred
to the next `run_frame()`, so old GameObjects accumulate in the registry's pending-destroy list
across rebuilds (no `run_frame` in tests). Pinned in `editor/test_scene_runtime_edges.py`.

### weather/classify.py::WeatherType — str() returns "WeatherType.X", not the value
Current: in Python 3.11 `Enum.__str__` overrides the `str` mixin, so `str(WeatherType.CLEAR)` is
`'WeatherType.CLEAR'`, not `'clear'`. `.value` and `==` comparisons still behave; any code doing
`str(weather_type)` expecting the plain value is silently wrong. classify() thresholds are all strict
`>` (boundary value falls to the lower category); STORM needs rain AND wind strictly. Pinned in
`test_weather_classify.py`.

### core/event_bus.py::EventBus — a raising handler aborts all later handlers (and drops drained events)
Current: `publish`/`drain` iterate `list(handlers)` with no per-handler guard, so one handler raising
propagates to the caller and skips every subsequent subscriber. In `drain()` the queue was already
swapped out, so the remaining deferred events are permanently lost. Documented as intentional
("exceptions propagate") but it makes one broken subscriber silently starve the rest. Unsubscribe-
during-publish IS safe (list copied first). Pinned in `test_event_bus_edges.py`.

### weather/lightning.py — load-resume verified SAFE (no finding)
`scheduled_strikes(t0,t2) == scheduled_strikes(t0,t1) + scheduled_strikes(t1,t2)` holds bit-exact
across grow/plateau/decay splits and multiple cells; `cell_id_int` is cross-process stable. No bug.

## Batch 3

### wind/modifiers.py::GustFront._phase_m — uses salted hash(), NOT cross-process deterministic (HIGH)
Current: phase offset is derived via `hash(repr(seed_key))`. Python's builtin `hash()` is salted per
process (PYTHONHASHSEED), so two processes constructing a `GustFront` with the SAME `seed_key` get
DIFFERENT phase offsets. The code comment claims "Pure: derived from the key text, not from any
process-salted hash" — but it IS salted. Violates Hard Rule 2 (determinism) for any cross-process
scenario (co-op, replay, save/load reproduce). In-process behavior pinned in `test_wind_modifiers.py`;
the cross-process divergence can't be pinned headlessly without a subprocess harness — flagged for fix.
Suggested fix: route through `core.rng.for_domain`/blake2b like the rest of the engine.

### weather/weather_map.py + weather/system.py::sample_fields — output depends on GLOBAL RNG at call time
Current: `WeatherSystem.sample_fields` (used by `WeatherMap.rasterize`) reads the global world seed at
SAMPLE time, not at construction — calling `set_world_seed(N)` after building a WeatherSystem changes
its subsequent output. In normal use the seed is set once at boot so this is benign, but it means
weather sampling is NOT a pure function of the instance; a refactor that changes when the seed is
captured could silently change behavior. Also `sample_fields` returns 5 channels (cov,den,rain,fog,
gust) but `rasterize`/`MAP_CHANNELS` keep only the first 4 (gust dropped — intentional but
undocumented). Pinned in `test_weather_map_grid.py`.

### weather/clouds.py — docstring/altitude-axis + magic-number notes (low severity)
Current: `CloudLayers.base_altitude_m` docstring says "increasing (low<mid<high)" but the stored array
is index 0=HIGH(1400m) … index 2=LOW(500m), i.e. decreasing by index (logical invariant still holds).
`classify_genus` docstring says "first match wins" but implementation is last-write-wins (same net
result). Cloud base altitude does NOT vary with precip/regime (only thickness does). Mid-band density
weights are hard-coded (not in config). Pinned/noted in `test_weather_clouds_edges.py`.

### weather/synoptic.py — displacement_vec named in docs but not implemented (doc/API mismatch)
Current: `weather.md` lists `displacement_vec(t_array)` but the module only exposes `displacement(t)`
(which already accepts scalar OR array). Importing `displacement_vec` would fail. dD/dt ≡ wind verified
consistent; negative t works unclamped. Pinned in `test_weather_synoptic_integral.py`.

### sky/celestial.py::moon_direction — dusk-only overlap vs docstring "both briefly visible"
Current: `MOON_PHASE_OFFSET_RAD` leads the moon's arc so sun+moon are both above the horizon at DUSK
(~17:30) but NOT at dawn (06:00 → moon z≈-0.25, below horizon). Docstring implies a general overlap.
Day-wrap is clean (modulo). Pinned (golden moon-z) in `test_sky_celestial_edges.py`.

### weather/cells.py::StormCell.active — both endpoints strictly exclusive
Current: `active(t)` is `0 < (t-spawn_time) < duration` — both spawn_time AND spawn_time+duration are
exclusive (existing tests only checked the start). `day_regime` accepts a float day and truncates to
int despite an `int` type hint. Pinned in `test_weather_cells_extra.py`.

## Batch 4

### TEST-QUALITY (ours): test_wind_modifiers.py band-travel assertion was flaky across processes
The first full-suite run failed `TestBandTravel::test_max_delta_shifts_with_time` (`63 != 63`) though
it passed in isolation. Cause: it asserted the discrete argmax CELL of the gust peak shifts over time,
but `GustFront._phase_m`'s process-salted hash (see Batch 3 finding) makes the absolute phase — hence
which cell the peak lands on — process-dependent; two times can share a cell. Fixed in-place to a
phase-independent assertion (the X-projection profile changes when the band advects). Pinned-behavior
intact; flakiness removed.

### EXISTING-TEST GAP: test_wind_venturi.py error-recovery test doesn't exercise a raising solve
While writing `test_wind_worker_edges.py` we found the existing worker error-recovery test feeds a
`dict` subclass overriding `.items()`, but `venturi.solve` never calls `.items()` (it uses
`materials.get(coord)`), so no exception is ever raised — and its single key is outside the test
region anyway. The existing test passes without actually testing error recovery. Our new file uses an
`ndarray` subclass with `__gt__` overridden, which genuinely triggers the worker's `except` handler.
Worth tightening the existing test.

### wind/region.py::WindRegion — floor-snap is asymmetric about zero; margin is strict `>`
Current: `_desired_origin` uses `int(np.floor(cell/snap_cells))*snap_cells`, so negative non-multiple
coords snap further negative (floor(-7/8)=-1 → origin -8, not 0). Recenter triggers only when distance
is strictly greater than `margin_cells*cell_m` (fence-post). Round-trip (out then back) restores the
same origin. Pinned in `test_wind_region.py`.

### wind/venturi.py::solve_venturi — wind_venturi_max clamp is effectively dead; solid cells get speedup>1
Current: `passw = clip(1-solid, 0.05, 1)` floors at 0.05, so `speedup = 1 + _CROWD_GAIN*crowd*passw`
maxes ~1.15 for any geometry reachable via VenturiJob — the configurable `wind_venturi_max` (2.2) is
never approached (dead knob). Fully-solid wall cells get a small speedup >1 (not 0, not "inside wall").
Pinned in `test_wind_venturi_edges.py`.

### wind/worker.py::VenturiWorker — stop() silently drops queued jobs; submit() after stop() leaks _pending
Current: on `stop()` the worker breaks on the sentinel and discards any still-queued jobs (their results
never drain, `_pending` stays positive). `submit()` before `start()` or after `stop()` does not raise
but inflates `_pending` permanently (no thread to decrement). A raising job posts an identity result
(all-ones speedup, zero deflect) and the worker survives. Pinned in `test_wind_worker_edges.py`.

### procedural/maps.py::derive_normal_map — alpha forced to 255; tangent sign convention
Current: output alpha is always 255 (input alpha ignored); a rightward-ascending luminance ramp yields
R<128 (normal tilts −X), B stays >128. Whether the shader expects this (OpenGL vs DX green) is a
render-side concern. Pinned in `test_procedural_maps.py`.

### procedural/flora/leaves.py::leaves_at_tips — leaf centers can land below ground (Z<0); fine-cell grid overflow
Current: no Z-floor clamp; the CA grid lower bound plus within-cell jitter pushes some leaf centers to
Z≈−0.7 m even when all branch tips are above ground. `cell_m=0.1` on a full-canopy oak trips the
`_MAX_GRID_CELLS` guard (`ValueError`); default 0.25 is fine. Pinned in `test_flora_leaves.py`.

### buildings/triangulate.py::triangulate_polygon — emits a zero-area triangle for collinear input; CW handling
Current: 0/1/2-vertex input → empty `(0,3)` uint32; collinear 3-point input emits a single zero-area
triangle (no raise/warn). CW input is reversed internally so emitted triangles are CCW and area is
preserved. Pinned in `test_buildings_triangulate.py`.

### lighting/exposure.py::ExposureMeter — "noon ≈ 1.0" is coincidental and adapts as a dark transition
Current: at clear noon the measured luminance ≈0.17988 → target ≈1.00067, which is just ABOVE the
current 1.0, so the meter uses the dark-adaptation tau (4.0 s, slow) and drifts up toward ~1.0007
instead of treating noon as neutral. The "stays near 1.0" behavior holds only by constant coincidence.
Pinned in `test_lighting_exposure.py`.

### devtools/gizmo.py::Gizmo.pick — axis handle dropped when ray origin lies on it; UNIFORM unpickable via centered ray
Current: `consider()` rejects a hit when `depth <= 0` (ray_s==0), so a ray whose origin sits exactly on
an axis handle misses it. In SCALE mode a ray through the pivot matches the X/Y axis handles at the same
depth, so the central UNIFORM handle can't be picked by a perfectly centered ray (the `update_drag`
math itself is fine). Pinned in `test_devtools_gizmo.py`.

## Batch 5

### core/math3d.py::Vec3.normalized / Quat.normalized — NaN passes the zero guard (silent NaN poison)
Current: the guard `length < 1e-12` is False for NaN (all NaN comparisons are False), so a vector/quat
with a NaN component does NOT raise — it divides and yields NaN components. Callers get silent NaN
poison rather than an error. Also: `length`/`length_squared` overflow float32→inf for components
≳1.84e19; `Quat.slerp` has no t-clamp (extrapolates outside [0,1], stays unit); `Quat.approx_eq`'s
eps boundary is effectively tighter than nominal because `_data` is float32. Pinned in
`test_math3d_boundaries.py`.

### player/fly_controller.py::FlyController — no constructor validation; NaN corrupts the transform
Current: `move_speed`/`sprint_mult`/`mouse_sensitivity` are `float()`-coerced with no validation —
NaN/inf/negative/zero all accepted; a NaN param silently corrupts `transform.local_position` on the
first movement frame. FlyController IS headless-importable (the `InputState` import is `TYPE_CHECKING`-
gated) — no refactor risk. Separately, `instantiate()` has no `name=` kwarg despite doc examples using
one (doc/API mismatch). Pinned in `test_fly_controller.py`.

### terrain/rain_cover.py::RainCoverField.recenter — no hysteresis; OPEN_SKY_Z float32 ≠ python -1e9
Current: `recenter` recomputes origin unconditionally every call (no threshold/hysteresis), so a
sub-cell move that crosses a grid-floor boundary shifts `origin_m`. `OPEN_SKY_Z` (-1e9) stored in the
float32 height array is `np.float32(-1e9) ≈ -999999987.0`, so exact `== -1e9` checks in caller code
silently fail (compare against `np.float32(OPEN_SKY_Z)`). `mark_dirty`/`rebuild` parity holds today but
the floor/ceil-vs-centre binning is a 1-texel-border risk under a future resolution change. Pinned in
`test_rain_cover_edges.py`.
